anatomy

Model

A mathematical function trained on data to make predictions or generate outputs from a prompt.

Two modes:

  • training — learning from vast amounts of text
  • inference — generating a response
Claude GPT Gemini Llama Mistral DeepSeek
  • This is an LLM — trained on lots of text; generating = "inference"
  • Precisely: its parameters (weights) are learned during training to predict the most likely next token
  • Runs locally or (usually) behind an API — Anthropic, OpenAI…
  • Not deep today: transformers, quality/speed/cost, open weights → future session
  • "Claude" = family; the model is the version (Opus 4.x, Haiku)

anatomy

Harness

The software system that wraps around a raw AI model, giving it tools, an execution environment, feedback loops, and guardrails.

chatbot→ copilot→ narrow agent→ general agentic

more autonomy, capability, and security risk →

Claude Code Cursor Windsurf ChatGPT Claude.ai Devin
  • Same model + different harness = very different behaviour
  • Impressive agent? ask what harness it runs
  • Both UI and CLI harnesses are on the slide
  • Spectrum: chatbot → copilot → narrow agent → general agentic
  • Capability ↔ security tradeoff runs along it

anatomy

Prompts

The input to a model, from which it generates a response.

  • User prompt — what you type.
  • System prompt — set by the harness, usually hidden from you.
text→ tokens→ embeddings→ neural network→ next token↻
  • Pipeline (below): text → tokens → embeddings → neural network → next-token (repeat)
  • Transformer = the neural-network architecture that weighs which earlier tokens matter most (attention) to predict the next
  • Transformers in depth: a future session
  • System prompt = invisible work; user prompt = your control
  • Iterate on the prompt — beats one-shotting it

anatomy

Tools

A function the model can call to fetch information or take an action beyond writing text.

read / write a file search the web run a command query a database call an API send an email control a browser

MCP — a standard way to connect tools to a model.
RAG — a tool that searches documents and feeds back what's relevant.

  • Base LLM: generates text · no access to outside info or systems
  • Tools = that access: data, files, APIs, email, a browser…
  • This is where a chatbot becomes an agent
  • MCP & RAG both live under "tools"

putting it together

Four questions for any agent.

1

What model is it calling?

2

What harness wraps it?

3

What's in the system prompt?

4

What tools can it access?

  • Agent acting weird? walk these four · most failures collapse to one
  • Our levers as designers, at each step:
  • — Model: swap the base model, or finetune it
  • — Harness: modify it, but often you can't access it
  • — System prompt: modify it, often can't access it
  • — Tools: add/remove; user-configurable to a point; build custom ones
  • Different products expose different config levels — each a tradeoff
  • Easiest → hardest: user prompt → swap base model (cost) → add/remove tools → alt harness/system prompt → finetune

how it runs

The agent loop.

1input a prompt 2the harness enriches it 3the model is called 4the model calls a tool 5the harness runs it, returns result to model 6repeat until model signals done
  • Click each step in — point at the box as it lights
  • 1 prompt → 2 harness enriches (system prompt + tools)
  • 3 model called → 4 model calls a tool
  • 5 harness runs it, returns result → 6 repeat until done
